优先优化方向：

1. 先建立统一的数据模型：WorkflowTemplate、AssetRecord、RunMeta、NodeOutput
2. 把工作流模板解析/保存/执行从 UI 里抽出去
3. 把 App.tsx 里和“节点派生、模板选择、资产预览、持久化”相关的逻辑拆成 hooks/services
4. 把临时调试脚本移动到 canvas/scratch 或删除不用的
5. 后端把 engine/src/app.ts 的业务逻辑拆进 services

一句话：
**现在代码能继续长，但如果不边做边收，后面会越来越难改。**
所以需要优化，但应该配合功能改造渐进做，不要先停下来大扫除。

我看到的主要问题是：

- canvas/src/App.tsx 太大，承担了太多职责：画布状态、弹窗、节点创建、模板选择、资产展示、持久化、事件分发都混在一起。
- 节点组件也偏重，比如图片、视频、TTS 节点里同时有 UI、状态、接口调用、模板逻辑。
- 有不少 inspect_*.cjs、output_*.txt、scratch 文件混在 canvas 根目录，像开发临时文件，后面最好归档或清理。
- engine 里已经有 Adapter 抽象，但部分接口还是直接在 app.ts 里写逻辑，后面应该服务化。
- RunningHub、本地 ComfyUI、普通 API 的概念还没有完全统一，所以前端会出现一些“这里叫 AIX，那里叫 RunningHub，那里又叫 custom workflow”的割裂。
- 配置、模板、资产、运行记录现在还没有形成统一数据模型。
