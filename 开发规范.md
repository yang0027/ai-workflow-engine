# Toonflow AI Workflow Engine 核心开发与改造规范

为了避免系统性路径问题、代码臃肿和难以调试的“黑盒”Bug，在接下来的开发改造中必须严格遵守以下规范。本规范已写入 AI 记忆脑区。

------

## 一、 开发核心原则

### 1. 功能分块与模块化（强约束）

任何新增或改造的节点与逻辑，必须进行物理拆分，禁止将 UI 渲染、业务逻辑、静态配置全部堆积在单个文件内。

- 单节点拆分结构：

  每个节点必须以文件夹形式组织（以CustomWorkflowNode为例）：

  ```
  nodes/CustomWorkflowNode/
  ├── index.ts                     # 统一导出入口
  ├── CustomWorkflowNode.tsx       # 仅负责 UI 渲染 (Handles、样式、静态布局、元素挂载)
  ├── CustomWorkflowNode.logic.ts # 仅负责业务逻辑 (getConnectedVal, handleExecute, 数据格式转换)
  └── CustomWorkflowNode.config.ts # 仅负责静态配置 (节点类型列表、输入输出端口定义、默认样式常量)
  ```

### 2. 强注释规范

任何模块、节点、关键数据转换函数（如 `getConnectedVal`）、适配器逻辑，必须添加详细的**中文注释**，重点写明：

- **输入输出格式**（例如：输入是 `minio` 还是 `base64` / `URL`，输出是何种结构）。
- **功能分块标识**（使用明显的分界注释，如 `// === 功能分块：IndexedDB 交互 ===`）。
- minio 端口 ：19000

## 九、新增节点标准规范

V2要加新节点时，按以下标准执行：

### 9.1 文件结构

```
src/components/nodes/节点名/
├── 节点名.tsx          ← UI渲染（只放JSX和样式）
├── 节点名.logic.ts     ← 业务逻辑（数据处理、API调用）
├── 节点名.config.ts    ← 配置（节点大小、Handle样式）
└── index.ts           ← 统一导出
```

### 9.2 必须用的钩子

| 钩子                | 用途        |  必须  |
| :------------------ | :---------- | :----: |
| useModelSelector    | 选厂家+模型 | ✅ 必须 |
| useWorkflowSelector | 选工作流    | ✅ 必须 |
| useParamMapper      | 参数映射    | ✅ 必须 |

### 9.3 数据格式标准

```typescript
// 输入
data.inputs = { prompt?, fileUrl?, refImages?, ... }

// 输出
data.outputs = { output, image?, video?, audio?, errorMsg? }
```

### 9.4 尺寸标准

所有节点默认 240x240，Handle位置统一（左进右出）。

### 9.5 存储标准

所有文件 → MinIO：http://localhost:19000/workflows/xxx
不用 db://，不用 IndexedDB。

- 完全对齐 `UploadNode` 逻辑。当用户在槽位上传文件时，通过 `UploadService.uploadBase64` 将生成的 base64 异步提交至 MinIO 网关（若网关繁忙则物理自愈降级转换为轻量的 `blob:http://...` URL）。
- **效果**：React 状态中只存储轻量级的 URL 地址（如 `http://localhost:19000/workflows/...`），零渲染延迟、零丢帧卡顿，彻底根治上传大图后的黑屏顽疾。
