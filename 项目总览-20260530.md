## 一、项目全景与核心架构

`ai-workflow-engine` 是一个基于 ReactFlow (`@xyflow/react`) 实现的高保真、智能节点化 AI 工作流配置与调度引擎。

### 1.1 系统运行环境

- **开发端**：React (Vite) + TypeScript (`npx tsc --noEmit --skipLibCheck` 保持零红线编译状态)。
- **运行端口**：Canvas 画布端 `5173`。
- **存储网关**：MinIO 独立容器，运行端口 `19000` (API网关) / `19001` (控制台)。

### 1.2 节点拓扑链与注册表

画布目前支持以下 **10 类基础与业务节点**（已注册在 `App.tsx` 的 `nodeTypes` 中）：

1. `prompt-source` (PromptSourceNode)：故事剧本源节点，支持高阶自由缩放。
2. `upload-node` (UploadNode)：本地媒体（图、音、视）上传。
3. `llm-service` (LLMStoryboardNode)：LLM 剧本分镜专家。
4. `image-service` (ImageServiceNode)：AI 智能画图生成。
5. `tts-service` (TTSServiceNode)：AI 声音克隆与文本转语音。
6. `video-fusion` (VideoFusionNode)：视频生成与智能合成 Fusion。
7. `purple-group` (PurpleGroupNode)：工作流分组容器。
8. `grid-splitter` (GridNode)：宫格智能多媒体排版。
9. `loop-node` (LoopNode)：参数或帧级的循环迭代执行器。
10. `custom-workflow` (CustomWorkflowNode)：自定义 ComfyUI 嵌套工作流。

## 二、节点标准四文件设计规范 (Four-File Structure)

为了最大化实现 UI 与业务逻辑的解耦，所有节点必须严格拆分到 `src/components/nodes/节点名/` 目录下，并以 4 个核心文件组织：

```
src/components/nodes/节点名/
├── index.ts              # 统一导出
├── 节点名.tsx            # 高保真 UI 渲染层 (无状态/极简状态)
├── 节点名.logic.ts       # 核心业务逻辑层 (自定义 Hook)
└── 节点名.config.ts      # 常量、元数据及 TypeScript 类型定义
```

### 2.1 4文件职责分工

#### 1) `index.ts` (导出文件)

负责暴露组件。内容极其精简：

```
typescript
export { default } from './节点名';
```

#### 2) `节点名.config.ts` (配置与类型)

声明节点配置、常量选项、默认数据及 TS 类型：

```
typescript
import { Node } from '@xyflow/react';
export interface 节点名NodeProps {
  id: string;
  data: {
    inputs?: Record<string, any>;
    outputs?: Record<string, any>;
    status?: 'idle' | 'running' | 'success' | 'error';
    [key: string]: any;
  };
  selected?: boolean;
}
export const DEFAULT_CONFIG = { ... };
```

#### 3) `节点名.logic.ts` (核心业务逻辑)

提取所有的 React hooks (`useState`, `useEffect`, `useCallback`, `useMemo`)、DOM 操作、API 请求、以及事件处理函数，导出为一个自定义的 `use节点名Logic` 钩子：

```
typescript
import { useReactFlow } from '@xyflow/react';
import { getUpstreamData } from '../../../hooks/getUpstreamData';
export function use节点名Logic({ id, data, edges, nodes }) {
  // 1. 读取上游
  const upstream = getUpstreamData(id, edges, nodes);
  // 2. 状态管理
  const [loading, setLoading] = useState(false);
  // 3. 执行核心业务
  const handleExecute = useCallback(async () => {
    setLoading(true);
    try {
      // 业务请求...
    } finally {
      setLoading(false);
    }
  }, [id]);
  return { loading, handleExecute, upstream };
}
```

#### 4) `节点名.tsx` (UI 渲染)

纯粹作为 UI 组件接收 Props，调用 `use节点名Logic` 钩子渲染视图，**禁止在 UI 文件中直接书写复杂的网络请求或长串业务处理函数**。

## 三、核心高阶 Hooks 规范

为保证不同节点的运行体验一致，所有节点在提取参数、获取模型、读取上游数据时，**必须且仅允许使用以下统一封装的高阶 React Hooks**。

### 3.1 `getUpstreamData` (统一读上游)

- **文件位置**：`src/hooks/getUpstreamData.ts`
- **规范说明**：任何节点不得通过手动筛选 `edges` / `nodes` 的方式跨级读取上游。必须通过该函数读取其直连的父级输出。
- **返回结构**：

```
typescript
function getUpstreamData(
  nodeId: string,
  edges: Edge[],
  nodes: Node[]
): {
  text: string[];   // 直连上游所有文本输出的合集
  image: string[];  // 直连上游所有图片输出的 URL/Base64 合集
  video: string[];  // 直连上游所有视频输出的 URL 合集
  audio: string[];  // 直连上游所有音频输出的 URL 合集
  all: any[];       // 上游原始 data 数据（用于高阶调试）
}
```

### 3.2 `useModelSelector` (统一模型选择)

- **文件位置**：`src/hooks/useModelSelector.ts`
- **规范说明**：当节点支持切换多厂家/多模型时（如 LLM 文本模型、Image 绘图模型），必须通过此组件绑定对应 capability，并在面板中唤起统一的模型弹窗。

### 3.3 `useWorkflowSelector` (统一工作流选择)

- **文件位置**：`src/hooks/useWorkflowSelector.ts`
- **规范说明**：ComfyUI 工作流、RunningHub 模板在节点内部引用时的统一调用钩子。

### 3.4 `useParamMapper` (统一参数映射)

- **文件位置**：`src/hooks/useParamMapper.ts`
- **规范说明**：将前端收集的 UI 表单数据映射为符合厂家 API 或者工作流的规范 JSON 参数体。

## 四、节点输入输出协议 (I/O Protocol)

各节点之间具备高度的数据流通性。必须保持强一致的数据契约。

### 4.1 输入协议 (`data.inputs`)

所有节点的外部输入信息（例如文本框内容、文件地址等）统一收纳在 `data.inputs` 中，且需支持连线上游时**自动同步与覆盖**。

```
typescript
data.inputs = {
  prompt?: string;       // 文本提示词
  fileUrl?: string;      // 关联文件（MinIO URL）
  refImages?: string[];  // 参考图合集
  // ... 其他节点专属参数
}
```

### 4.2 输出协议 (`data.outputs`)

为了使调度器、宫格排版和视频生成节点能够识别前置节点的生产要素，**所有节点输出必须严格遵从如下通用对象契约**：

```
typescript
data.outputs = {
  type: 'text' | 'image' | 'video' | 'audio'; // 核心输出的媒体格式
  output: string;                             // 核心输出的主要数据值 (URL 或文本内容)

 
  // 备用多媒体归宿管道 (当存在附带生成物时填充)
  text?: string;
  image?: string; 
  video?: string;
  audio?: string;
}
```

## 五、样式与物理交互规范 (Aesthetics & Interaction)

整个 Canvas 的设计遵循**暗黑科技（Dark-SciFi）、玻璃拟态（Glassmorphism）与紫色微光**的高端现代美学。

### 5.1 节点尺寸与样式基础参数

| 物理属性              |    规格要求     | CSS 实现参考                                   |
| :-------------------- | :-------------: | :--------------------------------------------- |
| **标准基准尺寸**      | `240px × 240px` | `width: 240, height: 240`                      |
| **容器圆角**          |     `16px`      | `border-radius: 16px`                          |
| **基础边框**          | 极细半透明灰白  | `border: 1px solid rgba(255, 255, 255, 0.08)`  |
| **选中边框**          |   亮紫色微光    | `border: 1.5px solid rgba(168, 85, 247, 0.85)` |
| **连接手柄 (Handle)** |  紫色中心圆环   | 24px 圆形，居中定位，层级 Z-Index 10           |

### 5.2 避坑指南：NodeResizer 拖拽冲突

- **问题现象**：在 ReactFlow 节点中启用缩放器 `NodeResizer` 时，如果最外层容器 `div` 包含 `custom-drag-handle` 样式，会导致点击边角缩放时触发整卡片拖拽，无法进行缩放。
- **唯一解决方案**：最外层容器 `div` **禁止添加 `custom-drag-handle`**。必须将 `custom-drag-handle` **精确且只绑定在卡片顶部的标题栏 (Header) 上**。仅在标题栏提供拖拽，其余区域留空，以便 `NodeResizer` 触发拉伸交互。

### 5.3 滚轮事件强阻断 (Wheel Event Capturing)

- **背景**：在节点内部放置多行文本框 (`textarea`) 时，如果用户在文本框内滚动滚轮，会导致整个 ReactFlow 画布缩放，文字无法正常滚动。
- **解决方案**：在 `WorkflowTextarea.tsx` 组件挂载时，通过 **原生原生捕获阶段 (Capture Phase)** 注入 wheel 监听器，并在轮询到有效滚动高度时执行 `e.stopPropagation()`，彻底切断向 ReactFlow 容器的事件冒泡：

```
typescript
textareaDom.addEventListener('wheel', (e) => {
  e.stopPropagation();
}, { capture: true, passive: false });
```

## 六、节点调度器与执行流 (Scheduler & Execution Flow)

### 6.1 `executeNode` 调度契约

当画布执行「步进执行」或「全画布执行」时，调度器将自底向上拓扑遍历，触发统一的执行动作：

```
typescript
async function executeNode(params: {
  nodeId: string;
  nodeType: string;
  upstreamData: ReturnType<typeof getUpstreamData>;
  modelConfig?: Record<string, any>;
  abortSignal?: AbortSignal;
}): Promise<{
  success: boolean;
  data?: any;
  error?: string;
  mediaType?: 'text' | 'image' | 'video' | 'audio';
}>
```

### 6.2 节点内部状态机流转

所有自定义节点必须根据 `data.status` 切换 UI 状态指示器：

- `idle` (空闲)：等待执行。
- `running` (执行中)：显示紫色旋转光圈、呼吸灯动效与当前执行进度条 (`data.progress`)。
- `success` (成功)：绿色微光边框。
- `error` (失败)：红色醒目闪烁边框，并把 `data.error` 展示在卡片底部。

## 七、存储与防灾自愈机制 (Storage & Self-Healing)

### 7.1 存储网关全面切换 MinIO

- 引擎已弃用历史版本中的前端高负荷 IndexedDB (`db://` 协议) 本地大资产写入机制。
- 所有的音视频、参考图在产出时，必须第一时间上传到远程 MinIO 存储网关，并转换为稳定的 `http://` 或 `https://` 实体 URL。
- **MinIO 默认Bucket**：`workflows`

### 7.2 历史数据防崩自愈 (Backward Compatibility)

为兼容历史画板导出的 JSON 文件，避免老用户的 IndexedDB 资产链接彻底损坏，系统在 `App.tsx` 与特定底层解析组件（如 `ResolvedMedia.tsx`）中保留了极简的**向下兼容自愈逻辑**：

- 如果解析到的数据值以 `db://` 开头，系统将智能提取其 `mediaId` 并通过全局挂载的 `window.getMediaFromDB(mediaId)` 临时转换回内存 Base64 显示，实现平滑自愈。

## 八、命名与代码提交约束 (Naming & Commit Rules)

### 8.1 命名规范

- **变量与核心函数**：驼峰命名法 `camelCase`（如 `getConnectedVal`）。
- **组件及类定义**：大驼峰命名法 `PascalCase`（如 `WorkflowTextarea`）。
- **本地存储 (LocalStorage)**：统一以 `canvas_` 作为 Key 前缀。
- **自定义 DOM 监听事件**：统一采用 `canvas-` 前缀且以中划线相连（如 `canvas-node-selected`）。
